# Ennegram v0.2 Section Index Design

## Purpose
Define section-level indexing so recall can return precise anchors instead of page `#top` summaries.

## Anchor Format (Locked)

- Canonical anchor id format:
  - `h2:<slug>`
  - `h3:<parent-slug>/<slug>`
  - `intro` for text before first heading
- Slug rules:
  - lowercase
  - alphanumeric + `-`
  - collapse whitespace/punctuation to `-`
  - trim leading/trailing `-`
- Duplicate heading disambiguation:
  - append `~2`, `~3`, ... by local occurrence order within the page.

Examples:
- `h2:chunking-implementation-code-truth`
- `h3:chunking-implementation-code-truth/shared-chunking-primitives`
- `h3:chunking-implementation-code-truth/shared-chunking-primitives~2`

## Markdown Section Split Rules (Locked)

Given parsed wiki page body (after frontmatter removal):

1. Create one synthetic `intro` chunk from start-of-body up to first heading.
2. Split on headings:
   - primary sections at `##`
   - sub-sections at `###`
3. Index units:
   - every `##` section becomes an index entry
   - every `###` section becomes an index entry
4. Section text boundaries:
   - chunk includes heading line + body until next heading of same or higher level.
5. Size guardrail:
   - if a section body exceeds 12k chars, split into numbered parts:
     - anchor suffix: `::part-1`, `::part-2`, ...
     - summary line remains heading text + `(part N)`.
6. Ignore headings deeper than `###` for v0.2 (retain in parent chunk).

## Ingest Output Shape (Compatibility)

Keep existing index fields and add section-level values:

- `id`: `<wiki-id>::<anchor>`
- `source_uri`: unchanged (wiki file path)
- `source_anchor`: section anchor (`intro`, `h2:...`, `h3:...`)
- `summary`: first non-empty line of section text (prefer heading line)
- `body`: section text only (not full page)
- `section_parent`: optional parent anchor for `h3` chunks
- `section_level`: `0` (intro), `2`, or `3`

Backward compatibility:
- continue indexing one legacy page-level `#top` entry per file in v0.2 rollout.
- recall can rank section entries first while preserving old behavior if section split fails.

## Tradeoffs (Brief)

- **Pros**
  - Better precision and clearer provenance in recall output.
  - Lower noise from generic page-level matches.
  - Incremental migration with compatibility preserved.
- **Cons**
  - Larger index size and potentially slower ingest.
  - More anchors to maintain in downstream tooling/docs.
  - Heading rename can affect anchor stability (mitigated by deterministic slug rules).

## Acceptance Checks for Day 3 Implementation

- Ingest report still succeeds with `errors=0`.
- Index contains section anchors beyond `top`.
- Recall displays section anchors in provenance for wiki hits.
- Existing commands and reports remain backward compatible.
